Pipr owns code host change request runtime behavior. Pi owns agent execution.

For a broader introduction, see [Pipr](/docs). For the public TypeScript API, see [Pipr SDK Reference](/docs/reference/sdk-reference).

## Package map

Each workspace package owns a distinct public or runtime boundary.

| Area | Responsibility |
| --- | --- |
| `packages/sdk` | Public TypeScript config builder, Run Bundle and review schemas, prompt helpers, and generated declaration source. |
| `packages/runtime` | Config loading, task orchestration, Diff Manifest creation, Pi execution, validation, publication planning, observability recording and archive retrieval, all code-host adapters, and the local adapter. |
| `packages/cli` | CLI command parsing and release binary entrypoint. |
| `packages/e2e` | Local Action fixture checks, container checks, and dry-run scenarios. |
| `apps/docs` | Fumadocs documentation app and migrated public docs content. |

The package root public surfaces stay small. Runtime internals stay behind domain folders such as `config`, `diff`, `pi`, `review`, `hosts`, and `action`.

## Runtime flow

The runtime keeps provider parsing and publication around a shared review pipeline.

1. Start the process Run Recorder before provider event parsing.
2. Parse the code host change request or command event through an adapter.
3. Load the trusted TypeScript config.
4. Build the Diff Manifest from local git state.
5. Run selected Review Tasks.
6. Call Pi through `ctx.pi.run()`.
7. Emit exactly one final output from the task.
8. For review output, validate structured findings against the Diff Manifest and build the Main Review Comment and Inline Review Comments.
9. For command response output, build a normal change-request comment keyed to the source command comment.
10. Publish comments through the code host adapter after stale-head checks.
11. Atomically finalize the Run Bundle, then let provider CI upload it.

## Data flow

Data moves from native event input to validated provider-neutral output.

```text
.pipr/config.ts
  -> RuntimePlan
  -> selected Review Task
  -> Diff Manifest
  -> Pi agent call
  -> structured ReviewResult
  -> validated Review Findings
  -> PublicationPlan
  -> code host adapter writes
```

The Diff Manifest is the central contract between git state, model-visible context, validation, and inline publication. Tasks should pass it into Pi instead of parsing raw diffs.

Run observability follows a separate provider-neutral path. `RunRecorder` captures lifecycle evidence while `RunArchiveSource` discovers and validates local or provider archives; neither interface expands `CodeHostAdapter`.

## Trust boundaries

For hosted change-request runs, Pipr loads `.pipr/config.ts` and local imports from the base commit. The proposed head is reviewed, but proposed config changes cannot change the current run's executable review settings.

Provider backend, model, API-key env, and provider options come from the trusted base config, not proposed change request code.

Pi receives a read-only workspace copy. The official Docker Action seals the copy under the root supervisor, then runs Pi as UID and GID `1000`. Pi can use only read-only built-in tools and Pipr-owned Diff Read Tools for condensed manifest runs.

## Extension points

User config can register models, agents, tasks, commands, limits, and typed plugin handles through `@usepipr/sdk`.

Pipr-owned runtime code keeps diff creation, Pi invocation policy, structured output validation, stale-head checks, marker dedupe, and code host writes out of userland tasks. GitHub, GitLab.com, Azure DevOps Services, and Bitbucket Cloud are supported.

See [Code Host Adapters](/docs/reference/code-host-adapters) for the adapter boundary.

## Review ownership

Review Tasks may:

- inspect provider-neutral change metadata
- request Diff Manifest projections
- call Pi with typed agents
- map custom output into comments
- emit one final review, command response, or check outcome

Review Tasks may not:

- write code host comments directly
- choose provider secrets at runtime from proposed change request code
- publish invalid inline ranges
- bypass stale-head checks
- mutate the Pi workspace

This split keeps `.pipr/config.ts` expressive without making it responsible for runtime safety.

## Adapter ownership

Core runtime produces provider-neutral events and publication items. Adapters map native provider behavior into that model.

The GitHub adapter owns:

- native event parsing
- permission normalization
- safe checkout
- comment API calls
- inline location payload mapping
- review thread reply and resolution APIs

Core runtime imports no GitHub modules directly.

## Config package layout

Pipr config supports two tiers:

- **Tier 1:** `.pipr/config.ts` only. No install step; runtime SDK stub supplies the module.
- **Tier 2:** `.pipr/package.json` + `.pipr/bun.lock` for third-party npm imports and self-contained editor types.

Action runs copy trusted `.pipr/**` from the base commit (excluding `node_modules`), project runtime-provided packages out of the copied lockfile, and reject any new or changed package tuple or metadata. Pipr then runs a frozen, integrity-verified `bun install` before replacing `@usepipr/sdk` with the image SDK stub for load and typecheck.
