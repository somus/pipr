The CLI is the local and Action entrypoint for Pipr.

```bash
pipr <command> [options]
```

Run `pipr <command> --help` for command-specific help.

## `pipr version`

Print the installed CLI version:

```bash
pipr version
pipr --version
```

The output is plain semver, such as `0.4.0`.

## `pipr update`

Update a CLI installed from the compiled GitHub Release binary:

```bash
pipr update
```

The command checks the latest release, downloads the matching binary only when a newer version exists, verifies the release checksum, validates the downloaded binary with `--version`, and replaces the current executable. If the installed version is already current, the command exits without replacing it.

`pipr update` does not update package-manager installs. For npm or Bun installs, use:

```bash
npm install -g @usepipr/cli@latest
bun install -g @usepipr/cli@latest
```

Compiled GitHub Release binaries are self-contained. The npm `@usepipr/cli` package executes through Bun and requires Bun 1.3.14 or newer.

`pipr update` updates only the local CLI executable. To update Pipr in GitHub Actions, change the workflow pin, such as `uses: somus/pipr@vX.Y.Z`.

## `pipr init`

Create editable TypeScript config:

```bash
pipr init
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--config-dir <dir>` | `.pipr` | Config directory to create. |
| `--adapters <adapters>` | `github` | Adapter files to initialize. Use `none` to skip workflow files. |
| `--recipe <recipe>` | none | Starter recipe id. |
| `--minimal` | false | Scaffold only `.pipr/config.ts` without `package.json` or `tsconfig.json`. |
| `--force` | false | Overwrite existing Pipr files. |

Examples:

```bash
pipr init --adapters none
pipr init --adapters gitlab
pipr init --recipe security-sast
pipr init --minimal
pipr init --config-dir .config/pipr
```

See [Init Recipes](/docs/recipes) for recipe ids.

## `pipr host-run`

Run a native change-request event through the selected Code Host Adapter:

```bash
pipr host-run --host github --event path/to/event.json
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--host <host>` | `PIPR_CODE_HOST` or detected CI host | Code Host Adapter id. |
| `--event <path>` | `PIPR_EVENT_PATH` or provider event path | Native event payload path when the provider supplies one. |
| `--config-dir <dir>` | `.pipr` | Config directory to load. |

The command selects a workspace from recognized provider CI variables, then falls back to the current directory. The GitHub Action invokes this command with `--host github` and preserves GitHub Actions logging and outputs. This release includes `github`, `gitlab`, `azure-devops`, and `bitbucket`.

## `pipr webhook serve`

Run a trusted, single-repository webhook worker with a durable SQLite queue:

```bash
PIPR_WEBHOOK_SECRET="..." pipr webhook serve \
  --host gitlab \
  --repository group/project \
  --workspace /srv/pipr/project \
  --database /var/lib/pipr/webhooks.sqlite
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--host <host>` | required | Webhook provider: `gitlab`, `azure-devops`, or `bitbucket`. |
| `--repository <repository>` | required | Expected GitLab project path/ID, Azure repository name/ID, or Bitbucket repository slug. Mismatched payloads are rejected. |
| `--workspace <path>` | required | Dedicated trusted repository checkout. |
| `--database <path>` | `.pipr/webhooks.sqlite` | Durable delivery and deduplication database. |
| `--hostname <hostname>` | `127.0.0.1` | Listen hostname. Keep loopback when using a reverse proxy. |
| `--port <port>` | `8787` | Listen port. |
| `--config-dir <dir>` | `.pipr` | Config directory to load from Trusted Base Config. |
| `--run-store-dir <path>` | `/var/lib/pipr/runs` | Diagnostic bundle store. |
| `--run-retention-days <days>` | `14` | Completed bundle retention. |
| `--run-max-bytes <bytes>` | `5368709120` | Maximum run-store size. |

`PIPR_WEBHOOK_SECRET` is required. Azure workers also require `AZURE_DEVOPS_ORGANIZATION`, `AZURE_DEVOPS_PROJECT`, and `PIPR_AZURE_SUBSCRIPTION_ID`. Bitbucket workers require `BITBUCKET_WORKSPACE`, `BITBUCKET_REPO_SLUG`, `BITBUCKET_EMAIL`, and `BITBUCKET_API_TOKEN`; command permission checks also require the separate workspace-admin permission credential documented in the [Bitbucket Cloud guide](/docs/guide/bitbucket). Provider credentials and model keys remain in the runner environment.

`GET /healthz` returns `200 OK` while the HTTP server can accept deliveries. It does not inspect provider credentials or run model work. See [Deploy the webhook runner with Docker](/docs/guide/webhook-docker) for the supported Compose deployment.

## `pipr webhook status`

Inspect recent results from a webhook worker's local SQLite queue:

```bash
pipr webhook status --database /var/lib/pipr/webhooks.sqlite
pipr webhook status --database /var/lib/pipr/webhooks.sqlite --limit 50 --json
```

The default table shows a bounded delivery ID, host, queue status, attempts, result kind, shortened Review Run ID, and update time. `--json` returns full delivery IDs and parsed safe Pipr Result V2 values in a versioned envelope. Results omitted by the 512 KiB per-result or 32 MiB aggregate retention limits report `size-limit` or `retention`; corrupted JSON reports `invalid` without hiding other deliveries.

| Option | Default | Purpose |
| --- | --- | --- |
| `--database <path>` | `.pipr/webhooks.sqlite` | Existing delivery database. Pipr fails without creating it when the path is missing. |
| `--limit <count>` | `20` | Number of deliveries to return, from 1 to 200. |
| `--json` | false | Print a versioned JSON envelope instead of the table. |

## `pipr runs list`

List local and provider records for one pull or merge request:

```bash
pipr runs list --pr 42
pipr runs list --pr https://gitlab.com/group/project/-/merge_requests/42 --json
```

| Option | Default | Purpose |
| --- | --- | --- |
| `--pr <selector>` | required | Change request number or provider URL. |
| `--host <host>` | Git remote or URL | Override the provider. |
| `--repository <repository>` | Git remote or URL | Override the provider repository path. |
| `--kind <kind>` | `all` | Filter `review`, `command`, `verifier`, or `startup`. |
| `--status <status>` | all | Filter by run outcome or artifact state. |
| `--limit <count>` | `20` | Return 1 to 1,000 records. |
| `--json` | false | Print a versioned JSON envelope, including partial-source errors. |
| `--store <path>` | `PIPR_RUN_STORE_DIR`, otherwise `.pipr-runs` and platform state | Select the local store merged with provider records. |

A numeric selector derives the host and repository from `remote.origin.url`. A provider URL works outside a checkout. Explicit `--host` and `--repository` values override discovery.

## `pipr runs show`

Diagnose one execution or the latest completed review for a change request:

```bash
pipr runs show 0123456789abcdef0123456789abcdef
pipr runs show --pr 42 --timeline
```

| Option | Default | Purpose |
| --- | --- | --- |
| `[execution-id]` | none | Select one 32-character execution trace ID. |
| `--pr <selector>` | none | Select the latest completed review for the change request. |
| `--host <host>` | Git remote or URL | Override the provider. |
| `--repository <repository>` | Git remote or URL | Override the provider repository path. |
| `--kind <kind>` | `review` with `--pr` | Select `command`, `verifier`, `startup`, or `all`. |
| `--timeline` | false | Print the complete span timeline. |
| `--json` | false | Print the manifest, spans, diagnosis, and artifact descriptors without prompt or output bodies. |
| `--store <path>` | `PIPR_RUN_STORE_DIR`, otherwise `.pipr-runs` and platform state | Select the local store merged with provider records. |

The diagnosis is deterministic and does not invoke an LLM.

## `pipr runs download`

Download and validate one bundle:

```bash
pipr runs download 0123456789abcdef0123456789abcdef --output ./pipr-run
```

| Option | Default | Purpose |
| --- | --- | --- |
| `<execution-id>` | required | Select one 32-character execution trace ID. |
| `--host <host>` | Git remote | Override the code host when downloading outside a checkout. |
| `--repository <repository>` | Git remote | Override the provider repository path. |
| `--output <path>` | `pipr-run-<execution-id>` | Write the validated unpacked directory. The destination must not exist. |
| `--archive` | false | Preserve the original provider archive beside the directory. |
| `--store <path>` | `PIPR_RUN_STORE_DIR`, otherwise `.pipr-runs` and platform state | Select the local store merged with provider records. |

See [Debug captured runs](/docs/guide/run-observability) for provider credentials, capture settings, artifact states, and bundle security.

## `pipr check`

Type-load config and validate the runtime plan:

```bash
pipr check
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--config-dir <dir>` | `.pipr` | Config directory to load. |
| `--require-env` | false | Fail when configured provider env vars are missing. |

Use `--require-env` when validating a local environment that should be able to run Pi.

## `pipr dry-run`

Load config and a native code host event without publishing:

```bash
pipr dry-run --host github --event path/to/event.json
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--host <host>` | `PIPR_CODE_HOST` or detected CI host | Code Host Adapter id. |
| `--event <path>` | required | Native event payload path. |
| `--config-dir <dir>` | `.pipr` | Config directory to load. |

Use this for Action event parsing and trusted config loading. Use `pipr review` for normal local review runs.

## `pipr inspect`

Print the loaded runtime plan:

```bash
pipr inspect
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--config-dir <dir>` | `.pipr` | Config directory to load. |

The output includes models, agents, tasks, commands, tools, publication settings, checks, and limits.

## `pipr skill`

Print or materialize the Pipr setup skill bundled with this CLI release:

```bash
pipr skill
pipr skill path
```

Subcommands:

| Command | Purpose |
| --- | --- |
| `pipr skill` | Print the bundled setup skill and its references to stdout. |
| `pipr skill path` | Materialize the setup skill under the local Pipr skill cache and print the directory path. |

The bundled skill is version-matched to the CLI. Use `pipr skill` when an AI agent needs to install Pipr, interview for repository review policy, create a custom `.pipr/config.ts`, and validate the setup.

## `pipr review`

Run configured change-request review tasks locally without publishing:

```bash
pipr review --base origin/main
```

Options:

| Option | Default | Purpose |
| --- | --- | --- |
| `--base <sha>` | required | Base commit, branch, or ref. |
| `--head <sha>` | working tree | Head commit, branch, or ref. Omit to include staged and unstaged tracked-file edits. |
| `--config-dir <dir>` | `.pipr` | Config directory to load. |
| `--pi-executable <path>` | `pi` on PATH | Pi executable for local runs. |
| `--trace [path]` | disabled | Capture a diagnostic bundle. A bare flag uses the platform Pipr state directory. |
| `--json` | false | Print structured local review output. |

See [Local runs](/docs/guide/local-runs) for behavior differences between local and hosted change-request runs.
