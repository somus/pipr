The Pipr SDK is the public TypeScript authoring API for `.pipr/config.ts`.

```ts
import { definePipr } from "@usepipr/sdk";
```

Type and schema names are exported from the SDK root:

```ts
import type { AgentTool, ReviewFinding, ReviewResult, ToolRunOptions } from "@usepipr/sdk";
```

## Concept map

The SDK maps directly to Pipr's runtime primitives. Use this table to choose the API surface before reading individual type details.

| Primitive | SDK surface |
| --- | --- |
| Entrypoint | `pipr.review(...)`, `pipr.on.changeRequest(...)`, `pipr.command(...)`, `defaultReviewEntrypoints` |
| Review Task | `pipr.task(...)`, `TaskContext` |
| Diff Manifest | `ctx.change.diffManifest(...)`, `DiffManifest`, `ReviewFinding.rangeId` |
| Agent | `pipr.agent(...)`, `ctx.pi.run(...)` |
| Review Output | `ctx.comment(...)`, `ctx.command?.reply(...)`, `CommentValue` |
| Publication | `pipr.config({ publication })`, Pipr runtime validation and code host adapters |

For the narrative model, see [How Pipr works](/docs/concepts).

## `definePipr`

Every config must default-export a synchronous factory:

```ts
import { definePipr } from "@usepipr/sdk";

export default definePipr((pipr) => {
  // register models, agents, tasks, commands, limits, tools
});
```

The callback must be synchronous. Runtime work belongs in `pipr.task(...)`, not at config load time.

## Models and secrets

Model ids default to `<provider>/<model>`:

```ts
const model = pipr.model({
  provider: "deepseek",
  model: "deepseek-v4-pro",
  apiKey: pipr.secret({ name: "DEEPSEEK_API_KEY" }),
  thinking: "high",
});
```

<auto-type-table path="packages/sdk/src/index.ts" name="ModelOptions" />

Pipr passes provider and model IDs to Pi. [Browse Pi's Model Catalog](https://pi.dev/models) for current identifiers, providers, context limits, and pricing. Use `id` only when you configure the same provider and model with different API keys or thinking levels. `pipr.secret({ name })` stores only the environment variable name. It does not read or serialize the secret during plan creation.

## Review recipe

`pipr.review(...)` is the quick structured review wrapper. It creates separate findings and summary agents, then wires their Review Task into change request and command entrypoints.

```ts
import { defaultReviewEntrypoints } from "@usepipr/sdk";

pipr.config({ publication: { maxInlineComments: 5 } });

pipr.review({
  id: "review",
  model,
  instructions: {
    findings: "Review the change for actionable correctness and security defects.",
    summary: "Summarize changed behavior, risk, and reviewer focus.",
  },
  tools: pipr.tools.readOnly,
  timeout: "10m",
  paths: {
    include: ["packages/runtime/**"],
    exclude: ["**/*.test.ts"],
  },
  entrypoints: {
    ...defaultReviewEntrypoints,
    command: {
      ...defaultReviewEntrypoints.command,
      description: "Run the default pipr review.",
    },
  },
});
```

`entrypoints.changeRequest` accepts these actions:

```ts twoslash
import { defaultReviewActions, defaultReviewEntrypoints } from "@usepipr/sdk";
import type { ChangeRequestAction } from "@usepipr/sdk";

const actions: readonly ChangeRequestAction[] = defaultReviewActions;
actions;
// ^?

defaultReviewEntrypoints;
// ^?
```

`entrypoints.command.permission` is provider-neutral:

```text
read < triage < write < maintain < admin
```

Adapters map native roles into this order.

To disable one entrypoint:

```ts
pipr.review({
  id: "review",
  model,
  instructions: {
    findings: "Review for actionable defects.",
    summary: "Summarize changed behavior and risk.",
  },
  entrypoints: {
    command: false,
  },
});
```

`paths` scopes the Diff Manifest and publishable Inline Review Comments for that review recipe. It is not a filesystem sandbox: Pi read-only tools can still inspect the repository, and Pipr prompts the agent to read nonmatching files only when needed to review matching files.

## Review prompts

The wrapper exposes separate stable instructions for findings and summary. Its built-in prompts are fixed so the runtime can supply the scoped Diff Manifest and merged findings consistently:

```ts
pipr.review({
  id: "review",
  model,
  instructions: {
    findings: "Prioritize broken public API contracts.",
    summary: "Explain the changed contract, risk, and reviewer focus.",
  },
});
```

Use `pipr.agent(...)` and `pipr.task(...)` when you need a custom per-run `prompt`. Do not repeat the change number, title, description, JSON output contract, or generic `suggestedFix` rules in instructions. Core supplies those sections once.

## Agents

Use `pipr.agent(...)` for non-default structured agent calls:

Agents have two prompt layers. Use `instructions` for stable behavior that applies to every run of the agent, such as role, review focus, and output constraints. Use `prompt` for the concrete task for one invocation. The `prompt` function receives the typed input passed to `ctx.pi.run(...)` and the agent prompt context, then Pipr renders it after the agent instructions in the Pi prompt.

```ts
const security = pipr.agent({
  name: "security-reviewer",
  model,
  instructions: "Review only security issues.",
  output: pipr.schemas.review,
  prompt: () => pipr.prompt`
    Review this change request for exploitable security issues.
  `,
  retry: {
    invalidOutput: 1,
    transientFailure: 1,
  },
  timeout: "10m",
});
```

`agent.extend(...)` creates a derived agent and appends additional instructions to the base instructions:

```ts
const strictSecurity = security.extend({
  instructions: "Ignore style-only findings.",
});
```

## Multi-agent orchestration

Use one task when multiple agents should produce one published review:

Pass `{ manifest }` to specialists and aggregators that need changed-code context. Pipr injects the authoritative Diff Manifest section from that reserved key. An aggregator that independently revalidates causality, impact, and anchors needs the manifest and read-only tools.

```ts
const base = pipr.agent({
  name: "base-specialist",
  model,
  instructions: "Return only actionable findings with valid ranges.",
  output: pipr.schemas.review,
  prompt: (input: { manifest: unknown; focus: string }) => pipr.prompt`
    ${pipr.section("Focus", input.focus)}
  `,
});

const security = base.extend({
  name: "security-specialist",
  instructions: "Focus on exploitable security issues.",
});
const tests = base.extend({
  name: "test-specialist",
  instructions: "Focus on missing regression tests.",
});

const aggregator = pipr.agent({
  name: "review-aggregator",
  model,
  instructions: "Revalidate and deduplicate specialist results into one review.",
  output: pipr.schemas.review,
  tools: pipr.tools.readOnly,
  prompt: (input: { manifest: unknown; specialistResults: unknown; prior: unknown }) => pipr.prompt`
    ${pipr.section("Prior review", pipr.json(input.prior, { maxCharacters: 20000 }))}
    ${pipr.section("Specialist results", pipr.json(input.specialistResults))}
  `,
});

const task = pipr.task({
  name: "multi-agent-review",
  async run(ctx) {
    const manifest = await ctx.change.diffManifest({ compressed: true });
    const prior = await ctx.review.prior();
    const [securityResult, testResult] = await Promise.all([
      ctx.pi.run(security, { manifest, focus: "security" }),
      ctx.pi.run(tests, { manifest, focus: "tests" }),
    ]);
    const result = await ctx.pi.run(aggregator, {
      manifest,
      specialistResults: { securityResult, testResult },
      prior,
    });
    await ctx.comment({ main: result.summary.body, inlineFindings: result.inlineFindings });
  },
});
```

## Tasks

Tasks are the executable review units. They receive a `TaskContext` and optional parsed input.

```ts
const task = pipr.task({
  name: "security-review",
  check: { name: "security" },
  async run(ctx) {
    const paths = { include: ["packages/runtime/**"] };
    const manifest = await ctx.change.diffManifest({ compressed: true, paths });
    const result = await ctx.pi.run(security, { manifest }, { paths });

    await ctx.comment({ main: result.summary.body, inlineFindings: result.inlineFindings });
  },
});

pipr.on.changeRequest({ actions: ["opened", "updated"], task });
pipr.command({ pattern: "@pipr security", permission: "write", task });
```

Local review uses tasks registered through `pipr.on.changeRequest(...)`. Set `local: false` on a task when it should be skipped by `pipr review` but still run in CI:

```ts
const ciOnlyTask = pipr.task({
  name: "ci-only",
  local: false,
  async run(ctx) {
    const manifest = await ctx.change.diffManifest({ compressed: true });
    const result = await ctx.pi.run(security, { manifest });
    await ctx.comment({ main: result.summary.body, inlineFindings: result.inlineFindings });
  },
});
```

Use `ctx.run.trigger === "local"` inside a task when the task should run in both environments but render different output. Keep `local: false` for tasks that should not run locally at all.

## Task context

Task handlers use this context to read change data, run Pi, publish results, and report status.

| Property | Purpose |
| --- | --- |
| `ctx.run.id` | Stable id for the selected Review Run, not the process attempt. |
| `ctx.run.trigger` | Provider-neutral task trigger: `change-request`, `command`, or `local`. `verifier` is reserved for Pipr's internal verifier context. |
| `ctx.platform.id` | Active code host adapter id. Don't use it to infer the Review Run trigger. |
| `ctx.repository` | Provider-neutral repository metadata. |
| `ctx.change` | Provider-neutral change request metadata plus diff helpers. |
| `ctx.pi.run(agent, input, options?)` | Execute a Pi-backed agent and validate structured output. |
| `ctx.command` | Present only for command-triggered tasks. Includes `name`, `line`, parsed `arguments`, and `reply(markdown)`. |
| `ctx.review.prior()` | Read prior review state: `main?`, `reviewedHeadSha?`, and `inlineFindings[]` with `id`, `status`, `path`, `rangeId`, `side`, `startLine`, and `endLine`. |
| `ctx.check.pass/fail/neutral(summary?)` | Set one explicit task check result. Check failure does not fail the Action process by itself. |
| `ctx.comment(value)` | Emit the selected run's final Main Review Comment markdown and Inline Review Comments. |
| `ctx.log` | Write runtime logs. |

`ctx.change.diffManifest(...)` returns commentable file ranges. Findings must reference those ranges by `rangeId`.

For languages supported by the installed ast-grep executable, Pipr can add `DiffManifestFile.changedSymbols` and `CommentableRange.summary` values derived from enclosing declarations. Caller-provided values take precedence. Derived values are best effort and bounded to 32 symbols per file, 120 characters per symbol, 200 characters per summary, and 32 KiB across one manifest. Compressed and automatically condensed projections retain symbols and summaries but continue to remove `signals` and range previews as applicable.

The Docker Action provides ast-grep `0.44.1`. Local runtimes use the same structural behavior when ast-grep is on `PATH`; otherwise, they return the ordinary Diff Manifest without failing the Review Run.

`ctx.command.reply(...)` emits a normal change-request comment response for command tasks. A task must emit exactly one final output: either `ctx.comment(...)` for review publication or `ctx.command.reply(...)` for command response publication.

When a custom task uses path scoping, pass the same `paths` to `ctx.change.diffManifest(...)` and `ctx.pi.run(...)`. The manifest is filtered to matching files, the prompt carries the path scope, and Pipr drops findings from that scoped Pi result when `result.inlineFindings` is passed directly to `ctx.comment(...)`. If a task maps, spreads, or clones the returned finding array, Pipr treats the transformed findings as task-authored output and applies normal Diff Manifest range validation.

## Code-host statuses

Task-level `check` controls individual code-host statuses and aggregate participation:

```ts
const task = pipr.task({
  name: "security",
  check: { enabled: true, name: "security", required: true },
  async run(ctx) {
    ctx.check.pass("Security review completed.");
    await ctx.comment("## Security review\n\nNo blocking findings.");
  },
});
```

When `check` is an object, the individual task status is enabled by default. `required` defaults to `true`. Use `check: false` to opt a task out of both its individual status and the aggregate status. A task without `check` does not get an individual status, but still participates in the aggregate when aggregate statuses are enabled.

Enable the aggregate status only when you want one Pipr-owned result across multiple tasks:

```ts
pipr.config({
  checks: { aggregate: { enabled: true, name: "all" } },
});
```

Statuses publish for supported change-request events when the selected adapter exposes status publication. GitHub uses Check Runs, GitLab uses external commit statuses, Azure DevOps uses pull request statuses, and Bitbucket uses source-commit statuses. They do not publish for command responses, local runs, or dry runs. `ctx.check.pass/fail/neutral(...)` can be called at most once per task. A failed task status does not fail the delivery process by itself.

## Global config

`pipr.config(...)` sets global runtime behavior:

```ts
pipr.config({
  publication: {
    maxInlineComments: 5,
    maxStoredFindings: 50,
    showHeader: true,
    showFooter: true,
    showStats: true,
    autoResolve: {
      model,
      instructions: "Respect maintainer explanations about deliberate product behavior.",
      synchronize: true,
      userReplies: {
        enabled: true,
        respondWhenStillValid: true,
        allowedActors: "author-or-write",
      },
    },
  },
  checks: {
    aggregate: { enabled: true },
  },
  limits: {
    timeoutSeconds: 300,
  },
});
```

`maxInlineComments` limits Inline Review Comments published by the current run. `maxStoredFindings` limits the current-first finding records serialized for future runs; it defaults to `50` and accepts values from `0` through `100`. Findings from an existing marker remain available during the run that rewrites it. Setting the limit to `0` writes no findings to the next marker, so subsequent runs receive no prior-finding input or automatic stale-finding resolution candidates; current output is unchanged.

`showHeader`, `showFooter`, and `showStats` independently control the generated Pipr title, attribution footer, and collapsed review statistics. All three default to `true`. Hidden publication markers and `PublicationResult.metadata.stats` remain available when their visible sections are disabled.

`publication.autoResolve` defaults to enabled. Use `autoResolve: false` to disable verifier-driven thread cleanup and user-reply handling. If `model` is omitted, Pipr uses the default provider. Use `instructions` to add project-specific guidance for the internal verifier.

`userReplies.allowedActors` controls who can trigger verifier replies:

| Value | Behavior |
| --- | --- |
| `"author-or-write"` | PR author or users with write access. Default. |
| `"write"` | Only users with write access. |
| `"any"` | Any commenter. Use only when public reply-triggered model calls are acceptable. |

`"author-or-write"` lets the PR author trigger verifier model calls from review-comment replies, including external contributors on public forks. Use `"write"` when only trusted repository collaborators should spend provider quota or send bounded diff, finding, and thread context to the configured model provider.

Set `respondWhenStillValid: false` to keep user-reply verifier runs silent unless the finding is resolved.

## Comment output

Each selected run must emit exactly one final output. Use `ctx.comment(value)` for review publication, or `ctx.command.reply(value)` for command response publication. Missing or duplicate final outputs fail the run.

Use `ctx.comment(...)` for Main Review Comment markdown:

```ts
await ctx.comment("## Security review\n\nNo exploitable findings.");
```

Use object form to publish a Main Review Comment, Inline Review Comments, or both:

```ts
await ctx.comment({
  main: "## Security review\n\nFound one issue.",
  inlineFindings: [{
    body: "This branch can throw before cleanup runs.",
    path: "src/server.ts",
    rangeId: "rng_...",
    side: "RIGHT",
    startLine: 42,
    endLine: 42,
  }],
});
```

For findings-only output, omit `main`:

```ts
await ctx.comment({ inlineFindings });
```

Structured output must contain `main`, `inlineFindings`, or both. `{ inlineFindings: [] }` is valid; `{}` is rejected.

Pipr validates findings against the Diff Manifest, drops invalid findings, dedupes finding markers, and caps inline publication.
Subjective labels such as severity, confidence, or category are not part of the built-in finding contract. Define them in a custom schema or render them into your own `ctx.comment(...)` markdown when your workflow needs them.

## Tools

`pipr.tool(...)` defines typed model-facing tools.

```ts
const lookupOwner = pipr.tool({
  name: "lookup_owner",
  description: "Return the owner for a repository path.",
  input: myInputSchema,
  output: myOutputSchema,
  async run({ input, ctx, signal }) {
    signal?.throwIfAborted();
    return await lookupOwnerFromCatalog(input.path, ctx);
  },
  toModelOutput(output) {
    return {
      owner: output.team,
    };
  },
});
```

Tool `run(...)` receives:

| Field | Meaning |
| --- | --- |
| `input` | Parsed tool input. |
| `ctx` | Runtime-provided tool context. |
| `signal` | Optional abort signal. |

Use `run({ input, ctx, signal })` for custom tools. Pipr rejects legacy
`execute(context, input)` tool definitions with `Tool '<name>' must define run`.

`toModelOutput(...)` can project a rich internal result into a smaller model-facing value.

## Schemas

Built-in schemas:

| Schema | Use |
| --- | --- |
| `pipr.schemas.review` | Main review result with summary and inline findings. |
| `pipr.schemas.inlineFindings` | Inline findings without a summary. |
| `pipr.schemas.summary` | Main-comment summary block. |

Custom schemas are useful for intermediate agents and workflows that map their final output into Pipr publication calls:

```ts
import { definePipr, z } from "@usepipr/sdk";

export default definePipr((pipr) => {
  const specialistOutput = pipr.schema({
    id: "security/specialist-output",
    schema: z.strictObject({
      summary: z.string(),
      risks: z.array(z.string()),
    }),
  });

  const summaryOutput = pipr.jsonSchema<{ summary: string }>({
    id: "security/summary",
    schema: {
      type: "object",
      additionalProperties: false,
      required: ["summary"],
      properties: {
        summary: { type: "string" },
      },
    },
  });

  void specialistOutput;
  void summaryOutput;
});
```

Use `pipr.schemas.review` when an agent directly returns publishable Inline Review Comments. `suggestedFix` is optional exact replacement code for the selected range. GitHub and GitLab render it as a native suggestion; Azure DevOps and Bitbucket render fenced replacement code.

Custom output receives Pipr's review policy automatically when its schema contains an object with the standard Review Finding fields. A schema rooted exactly at `{ inlineFindings: [...] }` is also automatically sharded and merged; each item may carry extra metadata. Nested, summary-only, and arbitrary schemas run once.

The `z` export is the recommended typed path for JSON-Schema-representable schemas. Generated `.pipr/types` include a standalone Zod authoring subset so `pipr init` projects type-check without installing `zod`; use `pipr.jsonSchema<T>()` for advanced JSON Schema shapes or when a Zod helper is outside that generated subset.

Parsing helpers and public Zod schemas are exported from `@usepipr/sdk`:

```ts
import {
  parseReviewFinding,
  parseReviewResult,
  reviewFindingSchema,
  reviewResultSchema,
  reviewSchemaExample,
  reviewSummarySchema,
} from "@usepipr/sdk";
```

Use `reviewResultSchema`, `reviewFindingSchema`, and `reviewSummarySchema` when you need the public review contract as Zod values.

Use prompt helpers to keep prompts structured:


`pipr.json(value, { pretty, maxCharacters })` renders bounded JSON and throws if the value is not fully JSON-serializable or exceeds `maxCharacters`.

## Runtime limits

Set global execution limits through `pipr.config(...)`.

```ts
pipr.config({
  limits: {
    timeoutSeconds: 300,
    maxAgentRuns: 16,
    diffManifest: {
      maxShards: 4,
      fullMaxBytes: 131072,
      fullMaxEstimatedTokens: 32000,
      condensedMaxBytes: 262144,
      condensedMaxEstimatedTokens: 64000,
      toolResponseMaxBytes: 65536,
    },
  },
});
```

These limits control runtime timeout, provider-call count, and Diff Manifest prompt sizing.

| Field | Values and default | Behavior |
| --- | --- | --- |
| `limits.diffManifest.maxShards` | Positive integer; defaults to `4` | Caps automatic fan-out from one review invocation. Set it to `1` to disable splitting. |
| `limits.maxAgentRuns` | Positive integer; unlimited when omitted | Caps total provider invocations across all tasks and agents in one Review Run. |

`limits.diffManifest.toolResponseMaxBytes` also bounds condensed helper responses. Condensed runs with structural capability register `pipr_read_declaration(path, ref, rangeId)` and `pipr_ast_grep(pattern, language, paths)` in addition to the manifest read tools when the reviewed head is the current workspace. Local ref-to-ref reviews with `pipr review --head <ref>` keep the standard read tools because the workspace may not match that ref. `pipr_ast_grep` searches only the head workspace, accepts 1 to 16 explicit safe paths, returns at most 100 matches, and truncates each matched text to 2 KiB.

`ctx.pi.run(agent, input, { maxShards })` overrides `limits.diffManifest.maxShards` for that invocation. Omit it to inherit the global value; set it to `1` to disable splitting for only that call.

Each shard, invalid-output repair, transient retry, and model fallback consumes one agent run. Pipr reserves an agent run immediately before starting Pi. The reservation remains consumed if the invocation fails or is aborted. After the budget is exhausted, Pipr fails before starting another provider process.

Increase `maxShards` if a capped condensed shard exceeds your provider's context limit. Higher values can make more provider calls, which also count against `maxAgentRuns` when configured.

## Plugins

Plugins install config-time handles through the builder:

```ts
import { definePlugin, z } from "@usepipr/sdk";

const owners = definePlugin((pipr) => {
  const input = pipr.schema({
    id: "owners/input",
    schema: z.strictObject({ path: z.string() }),
  });
  const output = pipr.schema({
    id: "owners/output",
    schema: z.strictObject({
      path: z.string(),
      owner: z.string(),
      policy: z.string(),
    }),
  });

  return {
    tool: pipr.tool({
      name: "owner_lookup",
      description: "Look up path owners.",
      input,
      output,
      run({ input }) {
        if (input.path.startsWith("packages/runtime/")) {
          return {
            path: input.path,
            owner: "runtime",
            policy: "Review publication behavior strictly.",
          };
        }
        return {
          path: input.path,
          owner: "general",
          policy: "Use the default repository policy.",
        };
      },
      toModelOutput(result) {
        return { path: result.path, owner: result.owner, policy: result.policy };
      },
    }),
  };
});

const ownerTools = pipr.use(owners);

const reviewer = pipr.agent({
  name: "owner-aware-review",
  model,
  output: pipr.schemas.review,
  tools: [...pipr.tools.readOnly, ownerTools.tool],
  instructions: "Use owner_lookup when ownership policy is relevant to the review.",
  prompt: () => "Review this change request.",
});

const ownerReview = pipr.task({
  name: "owner-aware-review",
  async run(ctx) {
    const manifest = await ctx.change.diffManifest({ compressed: true });
    const review = await ctx.pi.run(reviewer, { manifest });
    await ctx.comment({
      main: review.summary.body,
      inlineFindings: review.inlineFindings,
    });
  },
});

pipr.on.changeRequest({ actions: ["opened", "updated"], task: ownerReview });
```

Plugin setup runs during config loading. Runtime effects still belong in tasks and tools. Task, agent, and tool values are opaque handles; expose a separate plugin operation when task code needs callable behavior.
