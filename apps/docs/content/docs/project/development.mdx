Use this reference when you are changing Pipr itself. Run the focused check for the surface you changed, then run the repository gate before pushing.

## Setup

Install the pinned toolchain and run the repository gate before pushing broad changes.

### Install dependencies

Install the workspace dependencies and repository hooks through mise.

```bash
mise run install
```

This installs Bun workspace dependencies and hk Git hooks. The hooks run Biome before commits, check commit messages, and run the repository gate before pushes. If hooks are missing in an existing clone, run:

```bash
bun run hooks-install
```

Run the repository gate:

### Repository gate

Run the canonical repository check before pushing.

```bash
mise run check
```

This runs release metadata checks, script tests, package builds, lint, typecheck, tests, format check, and Fallow. It does not run the full docs Vite build.

## Docs app

Develop the Fumadocs app through its workspace scripts and keep generated output out of source edits.

### Local dev server

The docs app lives in `apps/docs` and is part of the Bun workspace and Turbo graph.

Run it locally:

```bash
bun run --cwd apps/docs dev
```

### Docs checks

Run docs-only checks:

```bash
bun run --cwd apps/docs typegen
bun run --cwd apps/docs typecheck
bun run --cwd apps/docs test
```

### Docs build

Run the full docs build only when validating docs deployment output:

```bash
bun run build:docs
```

Public docs content lives under `apps/docs/content/docs`. Generated files such as `.source`, `.output`, `.tanstack`, and `src/routeTree.gen.ts` are ignored.

When moving or adding pages:

### Page changes

Keep navigation, metadata, links, and generated output aligned when pages move.

- update the nearest `meta.json`
- keep frontmatter titles and descriptions useful
- use `/docs/...` links for internal docs routes
- keep source-only repo notes out of Fumadocs content
- run `bun run --cwd apps/docs typegen` before typecheck or build

## Local Action e2e

Use the Action gate for Docker packaging, event handling, workflow fixtures, and Pi CLI integration.

### Action gate

Use the local Action gate after editing the GitHub Action, Docker Action packaging, workflow fixtures, Pi CLI mapping, or PR event handling:

```bash
mise run check-actions
```

### What it runs

This command:

- builds the local Docker Action image as `pipr-action:act`
- verifies the installed Pi CLI contract
- runs fixture self-tests
- runs all local `act` scenarios

### Environment knobs

Supported environment knobs:

```bash
PIPR_ACTION_IMAGE=pipr-action:act
PIPR_SKIP_ACTION_IMAGE_BUILD=1
PIPR_ACT_RUNNER_IMAGE=catthehacker/ubuntu:act-latest
PIPR_ACT_PI_CALL_DIR=/tmp/pipr-pi-calls
```

Use `PIPR_SKIP_ACTION_IMAGE_BUILD=1` only when the target image already exists and should be reused.

## Prompt evals

Deterministic prompt eval smoke tests run with the e2e package tests:

```bash
bun run --cwd packages/e2e test
```

They use the fake Pi harness and do not call a model API. To point them at a custom fake Pi executable, set:

```bash
PIPR_EVAL_PI_EXECUTABLE=/absolute/path/to/fake-pi
```

To run only deterministic prompt eval smoke tests:

```bash
bun run --cwd packages/evals eval:deterministic
```

Live prompt evals run through Evalite and call DeepSeek through the local Pipr review path. Keep the key in the untracked `.pipr/.env`, then explicitly export only `DEEPSEEK_API_KEY` in a trusted shell. The package scripts do not auto-load the file because they execute code from the checked-out branch. Run them explicitly when model credentials are available:

```bash
bun run eval:prompts
bun run --cwd packages/evals eval:full
bun run --cwd packages/evals eval:full:export
```

`eval:prompts` runs focused live gates for recall, suppression, safety, and suggested-fix behavior. `eval:full` is the broader advisory suite, and `eval:full:export` writes `packages/evals/evalite-export/results.json`. Live evals are nondeterministic and are not part of `mise run check`.

## CI Docker e2e

The CI container lane exercises the built image directly without local `act` orchestration.

### CI container check

CI does not run `act`. The CI workflow builds a local single-platform Docker image with `load: true`, tags it as `pipr-action:e2e`, and runs:

```bash
PIPR_ACTION_IMAGE=pipr-action:e2e bun run --cwd packages/e2e check:container
```

`check:container` verifies the Pi contract and runs direct-container equivalents of the dry-run, full, condensed, and orchestrator scenarios.

The Dockerfile builds only the Action packages. Docs output is validated by the docs deployment path, not by the Action image build.

To run the same check locally:

### Local container check

Build and exercise the same container contract outside CI.

```bash
docker build -t pipr-action:e2e .
PIPR_ACTION_IMAGE=pipr-action:e2e bun run --cwd packages/e2e check:container
```

## Docker image release

Release workflows publish the verified Action image and retain immutable version tags.

### Action metadata

The checked-in Action metadata uses the published image directly:

```text
docker://ghcr.io/somus/pipr:v0.4.1 # x-release-please-version
```

Local `act` fixtures rewrite that metadata to the `PIPR_ACTION_IMAGE` value so Action checks still exercise the freshly built local image.

The Docker image is published only by the release workflow:

### Release workflow

Only the release workflow publishes the verified image.

```text
.github/workflows/release.yml
```

CI and the manual Docker workflow build local `pipr-action:e2e` images for verification only. They do not push to GHCR.

The GHCR image name is:

### GHCR image

All immutable and moving tags use this image name.

```text
ghcr.io/somus/pipr
```

### Published tags

Publishing uses the tags:

- release tag, such as `v0.4.1` {/* x-release-please-version */}
- release version, such as `0.4.1` {/* x-release-please-version */}
- `latest`

## npm and binary release

GitHub Releases and npm trusted publishing distribute the CLI and public packages from one release tag.

### Release artifacts

GitHub Releases publish two artifacts:

- compiled `pipr` CLI binaries attached to the release
- `@usepipr/sdk`, `@usepipr/runtime`, and `@usepipr/cli` published to npm

### Required setup

Required release setup:

- npm `@usepipr` package access
- `@usepipr/sdk`, `@usepipr/runtime`, and `@usepipr/cli` packages already created on npm
- npm trusted publishers configured for `somus/pipr`, workflow `release.yml`, allowed action `npm publish`
- GHCR package write access through `GITHUB_TOKEN`
- `PIPR_RELEASE_PLEASE_TOKEN` configured with permission to create release pull requests and releases

The Release Please workflow opens changelog and version pull requests from Conventional Commits on `main`. The publish workflow runs on `release.published`. The release tag must look like `v0.4.1`, and root, SDK, runtime, and CLI package versions must match the tag without the leading `v`. The workflow grants `id-token: write` and uses npm trusted publishing, so it does not need an `NPM_TOKEN` secret. {/* x-release-please-version */}

Configure trusted publishing through npm package settings, or with npm 11.15.0 or newer after the package exists:

### Trusted publishing

Configure npm's GitHub trusted publisher after the packages exist.

```bash
npm trust github @usepipr/sdk --repo somus/pipr --file .github/workflows/release.yml --allow-publish
npm trust github @usepipr/runtime --repo somus/pipr --file .github/workflows/release.yml --allow-publish
npm trust github @usepipr/cli --repo somus/pipr --file .github/workflows/release.yml --allow-publish
```

### Build binaries

Build release binaries locally:

```bash
bun run build:release:cli
```

Build only the host binary:

```bash
bun run build:release:cli --host --outfile /tmp/pipr
```

### Package dry runs

Check packages before publishing:

```bash
NPM_CONFIG_CACHE=/private/tmp/pipr-npm-cache npm pack --dry-run --json --workspace @usepipr/sdk
NPM_CONFIG_CACHE=/private/tmp/pipr-npm-cache npm pack --dry-run --json --workspace @usepipr/runtime
NPM_CONFIG_CACHE=/private/tmp/pipr-npm-cache npm pack --dry-run --json --workspace @usepipr/cli
```

## GitHub push checklist

Run the full gate, add the Action gate when required, and publish releases only from matching semver tags.

### Before pushing

Before pushing a public branch:

```bash
mise run check
```

### Action changes

For Action or Docker changes:

```bash
mise run check-actions
```

### Initial remote

Initial remote setup:

```bash
git remote add origin git@github.com:somus/pipr.git
git push -u origin main
```

### Draft release

Publish a GitHub Release from a semver tag when the package is ready:

```bash
gh release create v0.4.1 --draft --title v0.4.0 --generate-notes # x-release-please-version
```

The release workflow starts when the release is published.
