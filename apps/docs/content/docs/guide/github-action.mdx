The GitHub workflow runs `pipr host-run --host github` inside the prebuilt Pipr image. Pipr also supports [GitLab.com](/docs/guide/gitlab), [Azure DevOps Services](/docs/guide/azure-devops), and [Bitbucket Cloud](/docs/guide/bitbucket) through its provider-neutral runtime.

## Workflow

Use the generated workflow so events, checkout depth, caching, permissions, and secrets stay aligned with `pipr init`.

### Example workflow

The default workflow handles automatic reviews, commands, and verifier replies.

```yaml
name: pipr

on:
  pull_request:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]

permissions:
  contents: write
  pull-requests: write
  issues: write
  checks: write

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - uses: actions/cache@v4
        with:
          path: /home/runner/work/_temp/_github_home/.bun/install/cache
          key: pipr-bun-${{ hashFiles('.pipr/bun.lock') }}
      - uses: somus/pipr@v0.4.3 # x-release-please-version
        env:
          DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}
          GITHUB_TOKEN: ${{ github.token }}
```

## Events

The workflow listens only for events that can start configured tasks or verifier work.

| Event | Why Pipr listens |
| --- | --- |
| `pull_request` | Runs configured change request tasks on actions such as opened, synchronize, reopened, and ready for review. |
| `issue_comment.created` | Handles `@pipr` commands on pull requests. Comments on issues without a pull request are ignored. |
| `pull_request_review_comment.created` | Lets Pipr process replies to Pipr-owned Inline Review Comments through the verifier. |

The config still decides which tasks run. The workflow delivers events to `pipr host-run --host github`.

## Trusted options

The Action reads its trusted settings from workflow inputs and the base-commit config.

### Config directory

Use the default unless the repository deliberately stores Pipr config elsewhere.

| Option | Default | Purpose |
| --- | --- | --- |
| `config-dir` | `.pipr` | Directory containing `config.ts`. Local type support files are not required by the Action. |

Provider settings come from the base-commit `.pipr/config.ts`, not from pull request code. The Action uses the base config's default model and configured fallbacks, including provider backend, model name, API key env name, and provider options.

### Permissions

Use the generated permission block unless you have intentionally disabled the matching feature in `.pipr/config.ts`.

| Feature | GitHub write | Required permission |
| --- | --- | --- |
| Main Review Comment | Create or update a pull request issue comment | `issues: write` |
| Command response | Create or update the reply for one source `@pipr` command comment | `issues: write` |
| Inline Review Comments | Publish comments on valid diff ranges | `pull-requests: write` |
| Inline verifier replies | Reply to user comments on Pipr-owned Inline Review Comments | `pull-requests: write` |
| Review thread resolution | Resolve GitHub review threads for findings the verifier marks fixed | `contents: write` and `pull-requests: write` |
| Task or aggregate Check Runs | Create and update GitHub Check Runs when checks are enabled | `checks: write` |

Command help is exposed through the `main-comment` output. If GitHub denies verifier cleanup or the API fails, the review still succeeds and increments `result.publication.inlineResolutionErrorCount`.

### Secret redaction

The Action masks exact values for credentials Pipr receives at runtime, including sensitive environment variables and values resolved through `ctx.secret(...)`. Pipr applies this masking to Action logs and publication-bound review comments, suggested fixes, verifier replies, command responses, dropped-finding records, and Check Run summaries.

Pipr does not scan repository or model-generated text for unknown credentials. Use a repository secret scanner and revoke exposed credentials separately; Pipr's masking prevents known runtime credentials from creating an additional disclosure through logs or published output.

## Pinning

Pin the Action to the released version you intend to run.

### Release tag

Use the release tag you trust in production:

```yaml
- uses: somus/pipr@v0.4.3 # x-release-please-version
```

Use `somus/pipr@main` only when you intentionally want the current main-branch Action. The generated workflow uses the latest released tag at generation time.

## Outputs

The Action exposes one stable structured summary for diagnostics and integrations.

| Output | Description |
| --- | --- |
| `main-comment` | Rendered Main Review Comment body, command response body, or command help body. |
| `result` | Versioned JSON summary for review, ignored, dry-run, command-help, command-response, verifier, publication-error, and other fatal error outcomes. |

Parse `result`, require `formatVersion: 2`, then branch on `kind`. Review, command-response, and verifier results include a bounded `run` summary with reviewed refs, selected tasks, elapsed time, models, agent-run count, tokens, cost, and usage completeness. Review publication is `{ state: "completed", ...counts }`.

Pipr Result V2 does not expose raw errors, Diff Manifests, publication plans, internal markers or finding identity, native comment IDs, trusted configuration state, or provider credential metadata. V1 output is not supported.

## Trust model

GitHub pull request runs separate trusted base policy from proposed head code.

### Base branch config

For pull request Action runs, Pipr loads `.pipr/config.ts` and local imports from the pull request base commit. Pi reviews the pull request head, so PR-authored config changes are visible as code under review but cannot change review settings for the current run.

### Pi workspace

The root Pipr supervisor owns and seals a workspace copy, then starts Pi as UID and GID `1000`. Unix permissions prevent Pi from restoring write access to the copy. Pi receives only `read`, `grep`, `find`, and `ls`; condensed Diff Manifest runs may also receive Pipr Diff Read Tools over bounded diff data and base/head snapshots.

## Dry run

Dry runs validate event parsing and trusted config without model execution or publication.

### Action dry run

Set `PIPR_DRY_RUN=1` to load the event and trusted config without running Pi or publishing comments:

```yaml
env:
  PIPR_DRY_RUN: "1"
```

Dry run is useful for checking Action packaging, event parsing, and trusted config loading.

### Local event checks

For local event checks, use the CLI:

```bash
pipr dry-run --host github --event path/to/github-event.json
```

`dry-run` requires an event file because it exercises the Action event parser. For normal local review, use `pipr review --base <sha>`.

## Command comments

GitHub comment events dispatch configured commands and verifier replies.

### Pull request commands

`issue_comment` events are used for `@pipr` commands on pull requests. Pipr checks that the comment targets a pull request, fetches pull request metadata, checks the commenter permission, parses command arguments, and then starts the task.

Command tasks can answer with `ctx.command.reply(...)`. Pipr publishes that as a normal pull request issue comment and upserts it by source command comment.

### Verifier replies

`pull_request_review_comment` events are used for verifier replies. When a user replies to a Pipr-owned Inline Review Comment, Pipr can run its internal verifier against the current diff and thread context. If the finding is fixed, Pipr replies and resolves the thread. If the finding is still valid, Pipr can reply with a short explanation and keep the thread open.

### Permission order

Permissions are ordered:

```text
read < triage < write < maintain < admin
```

Command permission defaults to `write`.

## Forks

Fork reviews retain base-commit policy but follow GitHub's secret availability rules.

### Secret availability

GitHub decides which secrets are available for forked pull requests. If the provider secret is unavailable and the selected config requires it, Pipr fails before running Pi. Keep secrets in GitHub Actions secrets and reference them with `pipr.secret({ name: "ENV_NAME" })`; do not put raw keys in config.

### Base config on forks

The trusted config still comes from the base commit. A fork can propose changes to `.pipr/config.ts`, but those changes are reviewed as code and do not change the active run.
