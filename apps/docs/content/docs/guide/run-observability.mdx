---
title: "Debug captured runs"
description: "List, diagnose, and download Pipr run bundles from local and provider storage."
---

Pipr captures a diagnostic bundle for every hosted review, command, and verifier run. Use `pipr runs` to find slow phases, failed publication calls, invalid-output repairs, dropped findings, and missing evidence without rerunning the model.

Run bundles can contain redacted prompts, visible model output, diffs, and validation data. Treat downloaded bundles as sensitive repository data and share them only with people who can already access the change request.

## List runs for a change request

Pass a pull or merge request number from a repository checkout:

```bash
pipr runs list --pr 42
```

Pass the full provider URL when you are outside a checkout:

```bash
pipr runs list --pr https://github.com/owner/repository/pull/42
```

Pipr merges the selected local store with provider records and removes duplicates by execution ID. A run can report `available`, `available-in-ci`, `in-progress`, `expired`, `capture-failed`, `upload-failed`, `not-enabled`, or `indeterminate-missing`. `available-in-ci` means the provider UI has a native artifact, but its public API cannot download it; the list output links to the native CI run.

Set one retrieval credential for the selected provider. The credential needs artifact or build read access, not repository write access.

| Provider | Retrieval environment variable | Required access |
| --- | --- | --- |
| GitHub | `PIPR_GITHUB_TOKEN` or `GITHUB_TOKEN` | Actions read |
| GitLab | `PIPR_GITLAB_TOKEN` or `GITLAB_TOKEN` | Read pipelines, jobs, and job artifacts |
| Azure DevOps | `PIPR_AZURE_DEVOPS_TOKEN`, `AZURE_DEVOPS_TOKEN`, or `SYSTEM_ACCESSTOKEN` | `vso.build` |
| Bitbucket | `PIPR_BITBUCKET_TOKEN` or `BITBUCKET_API_TOKEN` | Read Pipelines and repository Downloads |

Pipr sends Azure PAT variables with Basic authentication and `SYSTEM_ACCESSTOKEN` as an OAuth Bearer token.

Use `--host` and `--repository` to override Git remote discovery:

```bash
pipr runs list --pr 42 --host gitlab --repository group/project --json
```

## Diagnose a run

Select the latest completed review for a change request:

```bash
pipr runs show --pr 42
```

Select a specific process by its 32-character execution ID:

```bash
pipr runs show 0123456789abcdef0123456789abcdef --timeline
```

The diagnosis is deterministic and does not call an LLM. It reports the critical path, phase durations, retries, repair attempts, time to first token, tool durations, token and cost usage, CPU and memory data, validation drops, publication failures, and missing evidence.

Use `--json` when another tool or coding agent will consume the diagnosis:

```bash
pipr runs show --pr 42 --json > pipr-diagnosis.json
```

The JSON response includes the manifest, spans, deterministic diagnosis, and artifact descriptors. It never includes prompt or model-output bodies; download the validated bundle when an authorized debugging process needs those files.

## Download a bundle

Download and validate an unpacked bundle:

```bash
pipr runs download 0123456789abcdef0123456789abcdef --output ./pipr-run
```

Outside a repository checkout, provide the provider coordinates explicitly:

```bash
pipr runs download 0123456789abcdef0123456789abcdef \
  --host github \
  --repository owner/repository
```

Add `--archive` to preserve the original provider archive beside the validated directory:

```bash
pipr runs download 0123456789abcdef0123456789abcdef \
  --output ./pipr-run \
  --archive
```

Pipr rejects unexpected files, path traversal, symlinks, invalid schemas, artifact hash mismatches, and archives that expand beyond 64 MiB.

## Capture a local review

Local reviews do not capture bundles unless you opt in. Pass a path to control the store:

```bash
pipr review --base origin/main --trace ./.pipr-runs
```

Pass `--trace` without a path to use the platform Pipr state directory, partitioned by repository:

```bash
pipr review --base origin/main --trace
```

The `pipr runs` commands search both `.pipr-runs` and this repository-partitioned platform state directory when neither `--store` nor `PIPR_RUN_STORE_DIR` is set.

Hosted runs use diagnostic capture by default. Operational flags and environment variables control capture independently from `.pipr/config.ts`:

| Setting | Default | Purpose |
| --- | --- | --- |
| `PIPR_RUN_CAPTURE` | `diagnostic` for hosted runs | Set `off`, `metadata`, or `diagnostic`. |
| `PIPR_RUN_STORE_DIR` | `.pipr-runs` in CI; `/var/lib/pipr/runs` for webhooks | Select the filesystem store. |
| `PIPR_RUN_RETENTION_DAYS` | `14` | Set webhook completed-run retention. |
| `PIPR_RUN_MAX_BYTES` | `5368709120` | Set the webhook store quota in bytes. |

Webhook cleanup removes expired runs first, then the oldest completed runs. A capture is protected only while its recorded process is still active, so a hard-killed process cannot leave a permanent quota exemption.

## Export content-free telemetry

Set standard `OTEL_EXPORTER_OTLP_*` and sampling variables to export traces, metrics, and logs over OTLP HTTP/protobuf. Pipr keeps local capture independent from `OTEL_TRACES_SAMPLER`, so disabling trace export does not discard bundle evidence.

OTLP data contains timings, statuses, sizes, hashes, usage, and provider-neutral run metadata. It excludes prompts, model output, source text, stderr snippets, environment values, raw reasoning, and tool payload bodies.

## Update existing CI configuration

Run `pipr init --force --adapters <provider>` only when you intend to regenerate all Pipr-owned starter files. Otherwise, copy the run-artifact step from the current generated provider example into the existing CI definition.

GitHub and GitLab keep bundles for 14 days. Azure publishes a pipeline artifact after the run. Bitbucket always captures a 14-day native Pipelines artifact and attempts a second upload to repository Downloads for automated retrieval. The Downloads upload requires separate secured `BITBUCKET_ARTIFACT_EMAIL` and `BITBUCKET_ARTIFACT_API_TOKEN` variables.
