Use this section when you are adding Pipr to a repository, changing how it reviews pull requests, or debugging what it publishes.

Pipr supports GitHub pull requests, GitLab.com merge requests, Azure DevOps Services pull requests, and Bitbucket Cloud pull requests.

Pipr is a lean core you build on: start with the default review, then compose recipes, custom tasks, commands, and plugins in the same `.pipr/config.ts` as your needs grow. The [Build on the core](/docs#build-on-the-core) ladder shows which page matches each level.

## First run

Follow this path for the first successful pull request review.

1. Install the CLI from the GitHub Release installer.
2. Run `pipr init` in the repository root.
3. Add the provider secret used by `.pipr/config.ts`.
4. Review the generated GitHub workflow and commit the files.
5. Open or update a pull request and check the Main Review Comment.
6. Use local review commands while tuning config.

## Setup success checklist

Use this checklist before changing reviewer instructions. It catches the most common first-run setup failures.

| Check | Command or place to inspect | Expected result |
| --- | --- | --- |
| CLI installed | `pipr --version` | A Pipr release version prints. |
| Config loads | `pipr check` | The runtime plan validates. |
| Plan matches intent | `pipr inspect` | Models, tasks, commands, and publication limits match `.pipr/config.ts`. |
| Provider secret exists | GitHub repository Actions secrets | The secret name matches `pipr.secret({ name })`. |
| Workflow has full history | `.github/workflows/pipr.yml` | `actions/checkout` uses `fetch-depth: 0`. |
| Workflow can write expected output | Workflow `permissions` block | Permissions match the enabled comment and check features. |
| First PR output appears | Pull request conversation and Files changed tab | Pipr publishes a Main Review Comment and any valid Inline Review Comments. |

If a check fails, use [Troubleshooting](/docs/guide/troubleshooting#first-run-failure-ladder) before changing task code.
