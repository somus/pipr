This guide gets Pipr running on a GitHub pull request. You will install the CLI, generate `.pipr/config.ts` and the GitHub workflow, add one provider secret, and confirm that Pipr can publish review output.

Pipr supports GitHub pull requests, GitLab.com merge requests, Azure DevOps Services pull requests, and Bitbucket Cloud pull requests. This quickstart uses GitHub; use the [GitLab guide](/docs/guide/gitlab), [Azure DevOps guide](/docs/guide/azure-devops), or [Bitbucket guide](/docs/guide/bitbucket) for those providers.

## First review checklist

Complete these checks in order. Do not tune prompts or recipes until the generated setup can run once.

| Milestone | Command or action | Success signal |
| --- | --- | --- |
| Install Pipr | Run the install command in step 1. | `pipr --version` prints a release version. |
| Generate config | `pipr init` | `.pipr/config.ts` and `.github/workflows/pipr.yml` exist. |
| Validate config | `pipr check` | The config loads without requiring a model call. |
| Add provider secret | Add `DEEPSEEK_API_KEY` or your configured secret | The workflow passes that env var to the Action. |
| Open a pull request | Commit the generated files and push a PR | The GitHub Action starts a Pipr run. |
| Confirm output | Open the PR conversation and Files changed tab | You see one Main Review Comment and any valid Inline Review Comments. |

## Prerequisites

Before you start, make sure the repository can run GitHub Actions and has a model provider key available.

- A GitHub repository with pull requests enabled.
- Permission to add a workflow under `.github/workflows`.
- Permission to add repository Actions secrets.
- A provider API key available as a GitHub Actions secret.
- A full checkout in CI, because Pipr compares pull request head against the trusted base commit.

## 1. Install Pipr

Install the local CLI before you generate repository config or run local checks.

### Install command

Install the CLI from the latest GitHub Release:

```bash
curl -fsSL https://pipr.run/install.sh | sh
```

### Release and install directory

The installer downloads the matching GitHub Release binary to `~/.local/bin`. Set
`PIPR_VERSION=v0.4.0` to install a specific release, or set `PIPR_INSTALL_DIR=/usr/local/bin`
to choose a different destination. {/* x-release-please-version */}

### Update Pipr

Update a CLI installed by the release installer:

```bash
pipr update
```

For package-manager installs, update the package instead:

```bash
npm install -g @usepipr/cli@latest
bun install -g @usepipr/cli@latest
```

Check the installed CLI version with:

```bash
pipr --version
```

Updating the local CLI does not update the GitHub Action used by a repository. To update CI, change the workflow pin, such as `uses: somus/pipr@vX.Y.Z`.

## 2. Create the default config

Generate the repository-owned review policy and validate it before adding secrets.

### Initialize and validate

Run `pipr init` from the repository root, then validate the generated config:

```bash
pipr init
pipr check
```

### Generated files

`pipr init` creates:

- `.pipr/config.ts`
- `.pipr/package.json` with `@usepipr/sdk`, `@types/bun`, and `typescript` for editor type support
- `.pipr/bun.lock` from the initial `bun install` (commit this file)
- `.pipr/tsconfig.json` for local editor type support
- `.pipr/.gitignore` so `.pipr/node_modules` stays out of git
- `.github/workflows/pipr.yml`

Existing Pipr files are not overwritten unless `pipr init --force` is used.

### Project layout

The resulting project layout is:

```text
repo/
├── .pipr/
│   ├── .gitignore
│   ├── bun.lock
│   ├── config.ts
│   ├── package.json
│   └── tsconfig.json
└── .github/workflows/pipr.yml
```

Use `pipr init --help` for alternate config directories, adapter generation, recipes, and the single-file `--minimal` layout.

## 3. Add the provider secret

Pass the model credential named by the trusted config through GitHub Actions secrets.

### Default model

The default config uses DeepSeek:

```ts twoslash
// @include: pipr-builder
const model = pipr.model({
  provider: "deepseek",
  model: "deepseek-v4-pro",
  apiKey: pipr.secret({ name: "DEEPSEEK_API_KEY" }),
});
```

### GitHub Actions secret

Create a GitHub Actions secret named `DEEPSEEK_API_KEY`.

If you change the model provider in `.pipr/config.ts`, commit that change and add the matching secret name. The Action reads provider env var names from the trusted base-branch config.

## 4. Review the workflow

Confirm that the generated workflow checks out full history, caches the Pipr package install, and grants the required permissions.

### Generated workflow

`pipr init` creates `.github/workflows/pipr.yml`:

```yaml
name: pipr

on:
  pull_request:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]

permissions:
  contents: write
  pull-requests: write
  issues: write
  checks: write

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - uses: actions/cache@v4
        with:
          path: /home/runner/work/_temp/_github_home/.bun/install/cache
          key: pipr-bun-${{ hashFiles('.pipr/bun.lock') }}
      - uses: somus/pipr@v0.4.2 # x-release-please-version
        env:
          DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}
          GITHUB_TOKEN: ${{ github.token }}
```

### Trusted checkout and config

`fetch-depth: 0` is required so Pipr can compare the pull request head against the trusted base commit. Pipr loads provider, model, and API key settings from the base-branch `.pipr/config.ts`, so pull request changes cannot alter the model used for that run.

### Workflow permissions

The generated workflow grants the permissions Pipr needs to publish comments, checks, and verifier thread updates. Use [GitHub Action](/docs/guide/github-action) for the full event, output, and permission details.

## 5. Commit and open a pull request

Commit the generated policy and workflow so the base branch can trust them on later reviews.

### Commit generated files

Commit the generated config and workflow:

```bash
git add .pipr .github/workflows/pipr.yml
git commit -m "Add Pipr review"
```

### Open the pull request

Open or update a pull request. Pipr runs the configured Review Task and publishes review output when the Action succeeds.

## 6. Trigger a manual review

The default config also registers an `@pipr` command for manual reruns. Comment on the pull request:

### Command comment

Create a new pull request comment with this exact command.

```text
@pipr review
```

### Command access

Command access defaults to repository `write` permission.

## Expected pull request output

A successful GitHub run publishes the review surfaces configured by `.pipr/config.ts`.

### Successful workflow run

GitHub Actions shows a successful `review` job after Pipr finishes the first pull request review.

/>

### Published surfaces

The default review publishes:

- one Main Review Comment
- Inline Review Comments for valid findings, capped by config
- optional Check Runs when checks are enabled by config
- Action outputs for the main comment, inline drafts, dropped findings, and publication metadata

### Dropped findings

If Pipr drops a finding, the finding failed runtime validation. The most common reason is an invalid path, side, or range for the current Diff Manifest. See [Comments and findings](/docs/concepts/comments).

## If the first run fails

Use the narrowest diagnostic first.

| Symptom | First diagnostic |
| --- | --- |
| Config fails before the Action runs | Run `pipr check`. |
| The runtime plan is not what you expected | Run `pipr inspect`. |
| The Action cannot compare the diff | Confirm `fetch-depth: 0` in the workflow. |
| The model does not start | Confirm the provider secret name and workflow `env` mapping. |
| The Action runs but does not publish | Check workflow permissions and publication metadata. |
| Inline comments are missing | Check the `dropped-findings` Action output. |

See [Troubleshooting](/docs/guide/troubleshooting#first-run-failure-ladder) for the full first-run ladder.

## Check the config locally

Use local commands to validate config and model behavior without publishing to GitHub.

### Local commands

Use local commands while tuning config or prompts:

```bash
pipr check
pipr inspect
pipr review --base origin/main
```

### Local review behavior

`pipr review` runs every unique change-request task from the local workspace config and prints output instead of publishing code-host comments. See [Local runs](/docs/guide/local-runs).

## Optional setup

Add recipes or alternate layouts only after the default review works.

### Init options

Use these options after the default review works:

- Run `pipr init --adapters none` to create only the `.pipr` config files.
- Run `pipr init --minimal` for a single `.pipr/config.ts` without `package.json`; install `@usepipr/sdk` at the repo root for editor types.
- Run `pipr init --recipe <id>` to start from a focused config such as `security-sast`, `multi-agent-review`, or `plugin-tool-review`.
