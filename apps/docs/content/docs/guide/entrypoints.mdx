title: "Entrypoints"
description: "Configure change request triggers, `@pipr` commands, local review, and permissions."
---

Entrypoints decide how a Review Task starts. A task can be selected by a change request event or an `@pipr` command comment. Local review runs the configured change-request tasks without publishing.

An Entrypoint only selects work. The selected Review Task still decides which Agents to run and which final output to emit.

## `pipr.review` entrypoints

### Default review entrypoints

The quickest setup sets a publication cap and uses `pipr.review(...)`:

```ts twoslash
// @include: pipr-reviewer
import { defaultReviewEntrypoints } from "@usepipr/sdk";

pipr.config({ publication: { maxInlineComments: 5 } });

pipr.review({
  id: "review",
  reviewer,
  entrypoints: defaultReviewEntrypoints,
});
```

### Disable an entrypoint

Set an entrypoint to `false` to disable it:

```ts twoslash
// @include: pipr-reviewer
pipr.review({
  id: "review",
  reviewer,
  entrypoints: {
    changeRequest: ["opened", "updated"],
    command: false,
  },
});
```

## Change request actions

### Supported actions

Supported public action names are:

```text
opened | updated | reopened | ready | closed
```

GitHub native actions are normalized into those provider-neutral names. For example, a pull request synchronize event maps to `updated`.

### Automatic review

Use change request triggers for automatic review:

```ts twoslash
// @include: pipr-task
pipr.on.changeRequest({
  actions: ["opened", "updated"],
  task,
});
```

## Commands

### Command registration

Commands run from pull request issue comments and must start with `@pipr`:

```ts twoslash
// @include: pipr-security-task
pipr.command({
  pattern: "@pipr security",
  permission: "write",
  description: "Run the security review task.",
  task: securityTask,
});
```

### Command response

Command tasks can publish a command response:

```ts twoslash
// @include: pipr-ask-agent
const ask = pipr.task<{ question: string }>({
  name: "ask",
  async run(ctx, input) {
    const manifest = await ctx.change.diffManifest({ compressed: true });
    const answer = await ctx.pi.run(askAgent, {
      question: input.question,
      manifest,
    });

    await ctx.command?.reply(answer.body);
  },
});

pipr.command({
  pattern: "@pipr ask <question...>",
  permission: "read",
  parse: ({ question }) => ({ question }),
  task: ask,
});
```

Command responses are normal pull request issue comments keyed to the source command comment. They do not update the Main Review Comment and do not publish Inline Review Comments.

Use `ctx.command?.reply(...)` only for command response workflows. Use `ctx.comment(...)` when the task should contribute review output to the Main Review Comment or Inline Review Comments.

## Command grammar

### Named captures

Use static words and named captures:

```ts twoslash
const pattern = "@pipr release-note <version>";
```

### Rest capture

Use a final rest capture for free-form text:

```ts twoslash
const pattern = "@pipr ask <question...>";
```

The rest capture must be the final required position. This keeps command parsing deterministic.

## Permissions

### Permission ladder

Permission levels are provider-neutral:

```text
read < triage < write < maintain < admin
```

GitHub repository permissions are normalized into these levels by the GitHub adapter. Command permission defaults to `write`.

### Lower-permission commands

Use lower permission only for commands that cannot publish review findings or expose sensitive context:

```ts twoslash
// @include: pipr-ask-task
pipr.command({
  pattern: "@pipr ask <question...>",
  permission: "read",
  task: ask,
});
```

## Local review

### Register local tasks

Local review runs without GitHub publication:

```ts twoslash
// @include: pipr-security-task
pipr.on.changeRequest({
  actions: ["opened", "updated"],
  task: securityTask,
});
```

### Run locally

Run it with:

```bash
pipr review --base origin/main
```

`pipr review` runs every unique task registered through `pipr.on.changeRequest(...)`. It ignores configured action filters because there is no local pull request event action.

### CI-only tasks

Set `local: false` on tasks that should not run locally:

```ts twoslash
// @include: pipr-security-agent
const securityTask = pipr.task({
  name: "security",
  local: false,
  async run(ctx) {
    const manifest = await ctx.change.diffManifest({ compressed: true });
    const result = await ctx.pi.run(securityAgent, { manifest });
    await ctx.comment({ main: result.summary.body, inlineFindings: result.inlineFindings });
  },
});
```

See [Local Runs](/docs/guide/local-runs) for base/head behavior.

## Event selection checklist

### Choose an entrypoint

| Need | Entrypoint |
| --- | --- |
| Automatic review on pull request updates | `changeRequest: ["opened", "updated"]` |
| Manual reviewer rerun | `command: { pattern: "@pipr review" }` |
| Free-form maintainer question | `pipr.command({ pattern: "@pipr ask <question...>" })` |
| Local preflight before pushing | Register change-request tasks and run `pipr review` |
| No automatic comments | Disable change request entrypoints and use commands only |
