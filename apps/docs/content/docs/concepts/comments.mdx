Pipr treats comments as runtime output, not userland side effects. Review Tasks emit one final value, then Pipr validates findings and publishes through the active code host adapter.

Use this page when a change-request comment is missing, duplicated, capped, or different from the model output.

Keep the layers separate: Agents return typed data to a Review Task, the task emits Review Output, and Pipr turns that output into published comments.

## Change-request output

Pipr can publish a review summary, validated inline findings, and command responses through the active adapter.

### Visible surfaces

A normal review can create three visible surfaces on a change request.

| Surface | Where it appears | What it is for |
| --- | --- | --- |
| Main Review Comment | Change-request conversation | The task summary, metadata, and non-inline workflow output. |
| Inline Review Comment | Change-request diff | One actionable Review Finding anchored to one validated Diff Manifest range. |
| Command response | Change-request conversation | The answer to one source `@pipr` command comment. |

### How to read a Pipr review

Start with the Main Review Comment, then inspect Inline Review Comments only for issues that need code changes.

| Part | What to look for | What it means |
| --- | --- | --- |
| Main Review Comment | Summary, counts, task metadata, and skipped or dropped finding notes | The overall review result for one task. |
| Inline Review Comment | One finding on one validated diff range | A concrete issue Pipr could anchor to changed code. |
| Suggested fix | Native suggestion or fenced replacement code | Exact replacement code for the selected range. |
| Dropped findings | Delivery output or metadata count | The model proposed a finding Pipr refused to publish. |
| Inline cap | Fewer inline comments than total findings | `publication.maxInlineComments` limited diff noise. |
| Prior state | Repeated finding ids and open locations | Pipr is tracking findings across reruns. |
| Verifier reply | A reply on a Pipr-owned inline thread | Pipr rechecked whether the finding is fixed or still valid. |

This shape keeps review finite: an optional summary, bounded inline issues, and explicit state for what changed across reruns.

### Debug outputs

GitHub Action outputs include a versioned structured summary with public inline findings, dropped findings, task checks, and sanitized publication counts and errors. See [GitHub Action outputs](/docs/guide/github-action#outputs) for the complete output contract.

## Task output types

Every selected task must emit exactly one of the supported final output shapes.

### Final output shapes

Tasks choose one final output shape. Pipr rejects missing or duplicate final outputs.

| Task output | Publication behavior |
| --- | --- |
| `ctx.comment(markdown)` | Publishes or updates the Main Review Comment only. |
| `ctx.comment({ main, inlineFindings })` | Publishes or updates the Main Review Comment and valid Inline Review Comments. |
| `ctx.comment({ inlineFindings })` | Publishes valid Inline Review Comments without a Main Review Comment. |
| `ctx.command?.reply(markdown)` | Publishes or updates a command response issue comment keyed to the source command. |

Structured comment output must contain `main`, `inlineFindings`, or both. `{ inlineFindings: [] }` is valid and publishes no comments; `{}` is rejected.

## Main Review Comment

The Main Review Comment carries the task's review-level result and stable Pipr state.

### Upserted summary

The Main Review Comment is the single visible summary for a review task. Pipr upserts it by hidden marker, then replaces the visible body on rerun.

/>

### Review stats

Expand **Review stats** at the bottom of a Main Review Comment to inspect the model work behind the review.

| Metric | Meaning |
| --- | --- |
| Models | Up to 20 concrete response models reported by Pi, or the requested model when Pi doesn't report one, in first-observed order. Additional model ids are omitted. |
| Agent runs | Every Pi process attempt, including retries, output repairs, fallbacks, and internal verifier runs. |
| Elapsed | Wall-clock Review Task runtime through task and verifier execution, sampled immediately before Pipr builds the publication plan. Parallel runs don't increase this value independently. |
| Input tokens | The sum of input tokens reported by completed Pi assistant messages. |
| Output tokens | The sum of output tokens reported by completed Pi assistant messages. |
| Cost (USD) | The sum of Pi's reported model costs. Pipr doesn't maintain a separate pricing table. |

Pipr labels usage values as **reported** whenever it can't safely aggregate complete usage, including missing runs or messages, invalid values, and numeric overflow. It shows **Unavailable** instead of zero when no usable usage event is available.

Command responses, skipped runs, and Main Review Comments that don't run Pi omit the section.

Set `publication.showStats` to `false` to hide this section. `publication.showHeader` and `publication.showFooter` similarly hide the generated Pipr title and attribution. Each option defaults to `true`, and hiding stats does not remove `PublicationResult.metadata.stats`.

#### Integration metadata

Integrations can read the same data from `PublicationResult.metadata.stats`:

| Field | Type | Meaning |
| --- | --- | --- |
| `models` | `string[]` | Up to 20 unique concrete or requested model ids in first-observed order. Additional ids are omitted. |
| `agentRuns` | `number` | Pi process attempt count. |
| `durationMs` | `number` | Elapsed wall-clock time in milliseconds. |
| `inputTokens` | `number` | Reported input-token total. |
| `outputTokens` | `number` | Reported output-token total. |
| `costUsd` | `number` | Reported cost total in US dollars. |
| `usageStatus` | `"complete" \| "partial" \| "unavailable"` | Whether usage covers every agent run. |

Use `usageStatus` when interpreting the numeric totals. `complete` covers every run and assistant message. `partial` contains only the safely aggregatable reported subset. `unavailable` means no usable usage event was available, so the token and cost fields are placeholders rather than observed zero usage.

### Main-only uses

Use the Main Review Comment for:

- review summary
- task metadata
- dropped finding counts
- reviewer context
- main-comment-only workflows such as PR briefing or changelog draft

Do not use it for every finding when the finding can anchor to a valid diff range. Inline comments are better for actionable code issues.

### Emit a main comment

Pass Markdown when the task has no inline findings to publish.

```ts twoslash
// @include: task-context
await ctx.comment("## Review\n\nNo blocking findings.");
```

## Inline Review Comments

Inline Review Comments connect actionable findings to validated changed-code ranges.

### Publish findings

Inline Review Comments come from `ReviewFinding[]`. Each finding must point to a valid commentable Diff Manifest range.

<img
  alt="GitHub inline review comment from github-actions bot with a native suggestion to change the transient failure threshold from 400 to 500"
  src="/images/pipr/providers/github-inline.png"
/>

```ts twoslash
// @include: task-result
await ctx.comment({
  main: result.summary.body,
  inlineFindings: result.inlineFindings,
});
```

### Finding shape

Each finding must target a valid Diff Manifest range:

```ts twoslash
import type { ReviewFinding } from "@usepipr/sdk";

const finding: ReviewFinding = {
  body: "The timeout is not applied to this subprocess call.",
  path: "packages/runtime/src/pi/runner.ts",
  rangeId: "rng_...",
  side: "RIGHT",
  startLine: 42,
  endLine: 42,
  suggestedFix: "await runWithTimeout(command, timeoutMs);",
};
```

### Suggested fixes

`suggestedFix` must be exact replacement code for the selected range. GitHub and GitLab publish native suggestions; Azure DevOps and Bitbucket publish fenced replacement code. Do not include Markdown fences, prose, labels, or partial context outside the replacement.

## Dropped findings

Pipr repairs a mismatched `rangeId` when the finding's path, side, and line span identify exactly one commentable range. It records ambiguous, invalid, or duplicate findings instead of publishing them at unverified locations.

### Drop reasons

Pipr drops invalid findings and continues the review. GitHub Action review results expose them as `result.droppedFindings`; the runtime also records their counts in internal publication metadata.

| Reason | Fix |
| --- | --- |
| `path` does not match a changed Diff Manifest file | Pass the manifest to the agent and ask for findings only on changed files. |
| `rangeId` does not exist and the path, side, and line span do not identify exactly one range | Use the `rangeId` values from the Diff Manifest. Pipr repairs the id only when the remaining anchor is unambiguous. |
| `side`, `startLine`, or `endLine` does not match the range | Copy the side and line range from the selected Diff Manifest range. |
| Finding is outside direct scoped Pi result paths | Use the same `paths` in `ctx.change.diffManifest(...)` and `ctx.pi.run(...)`, and pass `result.inlineFindings` directly when scope enforcement should apply. |
| Finding duplicates another current finding | Keep one finding per issue and range. |
| Output has unsupported fields | Use `pipr.schemas.review` or map custom schema output into `ReviewFinding[]`. |

### Review continues

Dropped findings do not fail the whole review because the remaining valid findings can still help reviewers.

## Inline caps

Cap inline comments to keep reviews useful and avoid flooding a change request.

```ts twoslash
// @include: pipr-builder
pipr.config({ publication: { maxInlineComments: 5 } });
```

The cap applies to published inline drafts. Keep the Main Review Comment useful when some findings are skipped by the cap, for example by summarizing counts or highest-risk findings.

## Prior state

Prior state lets Review Tasks and the verifier track Pipr-owned findings across reruns without changing what the current run publishes. Pipr encodes that state in hidden markers so later runs can compare findings from earlier heads.

### Stored fields

Pipr stores review state in hidden markers on Pipr-owned comments so reruns can keep track of prior findings.

State tracks:

- reviewed head SHA
- task scope
- finding ids
- normalized concern and selected-code fingerprints
- path and range
- side and lines
- status

Pipr stores at most 50 finding records by default. Set `publication.maxStoredFindings` from 0 to 100 to change that limit:

```ts twoslash
// @include: pipr-builder
pipr.config({ publication: { maxStoredFindings: 25 } });
```

Pipr stores current findings first, in publication order, then uses any remaining capacity for historical findings. This limit does not change the current run's finding count or Inline Review Comments.

Lower limits reduce the prior findings available to Review Tasks and the verifier on later reruns, including automatic stale-finding resolution candidates. On the first run after changing the limit, findings from the existing marker remain available and are rewritten to the new limit. Setting `maxStoredFindings: 0` writes no findings to the next marker, so subsequent runs have no finding-based cross-run behavior; other stored review metadata remains intact.

### Read prior state

Tasks can read prior state:

```ts twoslash
// @include: task-context
const prior = await ctx.review.prior();
```

Default review flows pass matching open findings back to the reviewer. Resolved-finding suppression is internal to Pipr and does not add fields to reviewer output or recipe schemas.

## Deduplication

Stable finding and location keys prevent reruns from creating repeated inline comments.

When a Pipr-owned native thread is resolved, Pipr records that status even if no Pipr resolution reply was posted. A later review suppresses a finding when its normalized concern text, path continuity, and selected source are an unambiguous match for one resolved finding. Changing the concern or selected lines makes it publishable again; nearby edits, line movement, and a file rename do not. Pipr publishes ambiguous matches instead of risking suppression of a distinct concern.

### Dedup keys

Pipr dedupes Inline Review Comments before publication.

Deduplication uses:

- stable finding id
- unique normalized concern and selected-code fingerprints with path continuity for resolved findings
- reviewed head SHA
- Pipr-owned same-head location overlap

This lets wording change without duplicating comments on the same range.

## Verifier replies

The internal verifier rechecks Pipr-owned findings after code changes or reviewer replies.

### Verifier triggers

The internal verifier can recheck Pipr-owned Inline Review Comments after new commits or reviewer replies.

The verifier can run after a new commit or when a user replies to a Pipr-owned Inline Review Comment. The active adapter maps the provider's native update and reply events into those triggers.

### Thread outcomes

When the verifier marks a finding fixed, Pipr can reply and resolve the native review thread when the adapter and permissions allow. When the finding is still valid, Pipr can reply with a short explanation and leave the thread open.

Thread cleanup failures increment the GitHub Action's `result.publication.inlineResolutionErrorCount` and do not make the review itself fail. The runtime keeps the detailed errors in internal publication metadata.

## Command responses

Command tasks answer the source comment without replacing the Main Review Comment.

### Source command responses

Command responses are separate from review comments. They are change-request comments keyed to the source command comment:

```ts twoslash
// @include: task-context
await ctx.command?.reply("The CI failure points to the package build step.");
```

After Pipr validates permission and command input, it creates or updates this comment with `accepted`, then `running`. A command that calls `ctx.command.reply(...)` replaces the same comment with the final response. A command that publishes a review ends with `completed`; failures end with `failed`, or `superseded` when the reviewed head changed.

Lifecycle status can update after head drift so the source command doesn't remain stuck on `running`. Pipr still stale-checks final model-generated responses and won't publish them for an outdated head. Rerunning the same source command updates the same Pipr-owned comment.
