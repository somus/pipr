Pipr runtime has one job: turn a code host change request into a deterministic review run.

Use this page for the end-to-end flow. Use [Comments and Findings](/docs/concepts/comments) for publication details and [Trust and Security](/docs/concepts/trust-security) for base config, secret, and workspace boundaries.

## Run modes

Choose the mode based on whether the run starts from a hosted event or local git state and whether it may publish.

### Mode matrix

The publication column distinguishes hosted review from local validation.

| Mode | Command or trigger | Publishes comments |
| --- | --- | --- |
| Change request event | Code host event through `pipr host-run` | Yes |
| Command comment | Native comment event such as `@pipr review` | Yes |
| Local review | `pipr review --base <sha> [--head <sha>]` | No |
| Dry run | `pipr dry-run --host <host> --event <path>` or `PIPR_DRY_RUN=1` in a hosted run | No |

Local review uses the local code host adapter. It builds the same Diff Manifest and executes every unique change-request task that is not marked `local: false`, but publication is disabled.

See [Local runs](/docs/guide/local-runs) for local command examples.

## Change request context

Runtime tasks receive the same repository and change metadata regardless of code host.

### Task context shape

Runtime tasks receive provider-neutral context:

```ts
import type { DiffManifest } from "@usepipr/sdk";

type TaskContext = {
  platform: { id: string };
  repository: {
    root: string;
    owner?: string;
    name: string;
    defaultBranch?: string;
    remoteUrl?: string;
  };
  change: {
    number?: number;
    title: string;
    description: string;
    url?: string;
    author?: { login: string };
    base: { ref?: string; sha: string };
    head: { ref?: string; sha: string };
    isFork?: boolean;
    diffManifest(options?: {
      compressed?: boolean;
      paths?: {
        include?: string[];
        exclude?: string[];
      };
    }): Promise<DiffManifest>;
  };
};
```

`change.number` is provider-relative: a GitHub, Azure DevOps, or Bitbucket Cloud pull request number, or a GitLab merge request IID.

## Trust model

Hosted change-request runs follow this order:

1. Pipr loads `.pipr/config.ts` and local imports from the base commit.
2. Pipr checks that configured provider env vars are available.
3. Pipr checks out the exact proposed head for review.
4. Proposed changes to `.pipr/config.ts` are reviewed as code, but do not affect the current run.
5. Pi receives a read-only workspace copy and Pipr-owned diff tools. The official Docker Action enforces the copy across separate supervisor and Pi identities.

This keeps model choice, API-key env names, task registration, command permissions, and publication policy out of untrusted proposed code.

### Trusted policy

See [Trust and Security](/docs/concepts/trust-security) for a focused breakdown.

## Runtime flow

The adapter handles native event and API details around a provider-neutral review pipeline.

### Execution pipeline

Every provider follows this sequence after its adapter accepts an event.

```text
code host event
  -> adapter parses provider event
  -> trusted config loads
  -> adapter ensures checkout
  -> Diff Manifest builds
  -> selected tasks run
  -> Pi returns structured JSON
  -> pipr validates findings
  -> publication plan reduces task output
  -> adapter publishes native comments
```

### Adapter boundary

Core runtime never calls provider APIs directly. It receives a `ChangeRequestEventContext` and delegates provider-specific reads and writes to the active adapter.

## Diff Manifest

The Diff Manifest is the authoritative changed-code model for prompts, path scope, and inline validation.

### Build manifest

The Diff Manifest is the source of truth for changed files and commentable ranges:

```ts twoslash
// @include: task-context
const manifest = await ctx.change.diffManifest({
  compressed: true,
  includePreviews: true,
  paths: {
    include: ["packages/runtime/**"],
    exclude: ["**/*.test.ts"],
  },
});
```

### Finding anchors

Each file carries changed-line ranges that can receive inline comments. Review Findings must point at a valid range:

```ts twoslash
import type { ReviewFinding } from "@usepipr/sdk";

const finding: ReviewFinding = {
  body: "Specific issue and why it matters.",
  path: "src/example.ts",
  rangeId: "rng_...",
  side: "RIGHT",
  startLine: 12,
  endLine: 12,
  suggestedFix: "return safeValue;",
};
```

`suggestedFix` is optional exact replacement code for that range. GitHub and GitLab render native suggestions; Azure DevOps and Bitbucket render fenced replacement code. Do not include Markdown fences, prose, or labels in the value.

### Prompt injection

Diff Manifest prompt injection is reserved opt-in. When `ctx.pi.run(...)` receives a parseable Diff Manifest in the `manifest` input field, Pipr injects the authoritative Diff Manifest section exactly once. If `manifest` is omitted, no Diff Manifest section is injected. If `manifest` is present but not a parseable Diff Manifest, it stays ordinary user input.

### Condensed reads

When the full manifest is too large, Pipr sends a condensed manifest and attaches bounded Diff Read Tools. The model can ask for more range-scoped context without receiving shell or arbitrary filesystem access. If an agent runs with `toolMode: "none"`, Pipr still injects manifest prompt context when `manifest` is passed, but it does not attach Diff Read Tools.

### Path scope

`paths` filters Diff Manifest files by include and exclude globs. For renamed files, current and previous paths are considered. When `result.inlineFindings` from a scoped Pi run is passed directly to `ctx.comment(...)`, Pipr drops findings outside the filter. Findings that a task maps, spreads, or clones first are treated as task-authored output and receive normal Diff Manifest range validation. Path scope does not restrict Pi read-only access to the rest of the repository.

## Pi execution

Tasks invoke Pi through the runtime so model selection, retries, tools, and output validation stay controlled.

### Run agents

Tasks call Pi through `ctx.pi.run(...)`:

```ts twoslash
// @include: task-pi-review
const result = await ctx.pi.run(
  reviewer,
  { manifest },
  {
    timeout: "5m",
    fallbacks: [backupModel],
  },
);
```

Core adds the change request number and title, plus up to 4,000 characters of its description when nonblank, to every agent prompt as untrusted intent context. Pass only task-specific input here; do not repeat change metadata in the prompt.

### Model selection

Model selection order:

1. `ctx.pi.run(...)` call override
2. `agent.model`
3. `agent.fallbacks`
4. the trusted base config default provider

### Output repair

Invalid structured output receives one repair attempt when configured. Transient retries are opt-in through agent retry settings.

## Publication

Tasks emit one final value, then Pipr validates and maps it to native provider output.

### Final outputs

Review Tasks do not write code host comments. A selected run emits one final output:

- `ctx.comment(markdown)`
- `ctx.comment({ main, inlineFindings })`
- `ctx.command.reply(markdown)` for command-triggered response tasks

### Publication plan

For review output, Pipr turns that output into a provider-neutral `PublicationPlan`:

- one Main Review Comment
- zero or more Inline Review Comment drafts
- dropped finding metadata
- run metadata

Command response output publishes a normal change-request comment keyed by the source command comment. Rerunning the same source command updates the previous bot response.

### Comment updates and state

Before publishing, Pipr checks the current head SHA. It upserts the Main Review Comment by hidden marker and replaces the visible main body wholesale. That marker also carries Pipr-owned review state for prior finding IDs, locations, statuses, task scope, head metadata, and normalized selected-code fingerprints. Reruns pass matching open findings back to default review agents. During publication, Pipr suppresses a resolved finding only when its selected code uniquely matches one prior resolved record; this applies to every recipe that publishes `ReviewFinding` values and requires no reviewer-managed identity field. Pipr also keeps open findings visible in state unless the internal verifier marks them fixed. Native update and reply events can recheck prior Pipr Inline Review Comments, reply with the resolving commit, and resolve the provider thread when the adapter and permissions allow. Cleanup failures are recorded in publication metadata without failing the review. Inline Review Comments are deduped by stable finding ID, unique resolved-finding selected-code fingerprint, reviewed head SHA, and Pipr-owned same-head location overlap.

See [Comments and Findings](/docs/concepts/comments) for the publication model.

## Failure behavior

Failures stop at the narrowest safe boundary and preserve publication metadata when available.

### Failure matrix

The runtime records or stops each failure according to whether safe publication can continue.

| Failure | Behavior |
| --- | --- |
| No matching event or command | Delivery is ignored. |
| Config load or validation failure | Run fails before Pi execution. |
| Missing provider env when required | Run fails before Pi execution. |
| Pi returns invalid JSON after repair | Task fails and metadata records failure. |
| Finding targets invalid range | Finding is dropped, run continues. |
| Stale head before publication | Publication stops before writes. |
| Comment API failure | Publication metadata is emitted and the hosted run fails. |
| Stale review-thread cleanup failure | Publication metadata records the cleanup error and the Action continues. |

## Local checks

Run these commands before testing hosted delivery.

### Preflight commands

Run config validation before invoking a model locally.

```bash
pipr check
pipr inspect
pipr review --base origin/main
```

`pipr inspect` prints models, agents, tasks, change request triggers, commands, tools, publication settings, and limits.
